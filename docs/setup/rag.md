# RAG Memory (Vector Embeddings)

VeriClaw supports retrieval-augmented generation (RAG) by storing message embeddings in a sqlite-vec virtual table and exposing a `memory_search` tool that lets the agent semantically recall relevant past conversations.

## Architecture

1. **sqlite-vec** – a SQLite extension that adds KNN vector search via `vec0` virtual tables.
2. **text-embedding-3-small** – embeddings are generated by calling `POST /v1/embeddings` on the configured provider endpoint (OpenAI-compatible).
3. **`memory_search` tool** – when enabled, the agent can call this tool to retrieve the top-K most semantically similar stored memories.

## Installing sqlite-vec

### macOS (Homebrew)

```bash
brew install sqlite-vec
```

After installation, the shared library is typically at `/opt/homebrew/lib/vec0.dylib` or available as `vec0` on the dynamic linker path.

### Linux (from releases)

Download the latest release from https://github.com/asg017/sqlite-vec/releases and install the shared library:

```bash
# Example for x86_64 Linux
wget https://github.com/asg017/sqlite-vec/releases/download/v0.1.6/sqlite-vec-0.1.6-loadable-linux-x86_64.tar.gz
tar -xzf sqlite-vec-*.tar.gz
sudo cp vec0.so /usr/local/lib/
sudo ldconfig
```

### Docker

Add to your `Dockerfile.release`:

```dockerfile
RUN apt-get install -y libsqlite3-dev && \
    wget -q https://github.com/asg017/sqlite-vec/releases/download/v0.1.6/sqlite-vec-0.1.6-loadable-linux-x86_64.tar.gz \
    -O /tmp/sqlite-vec.tar.gz && \
    tar -xzf /tmp/sqlite-vec.tar.gz -C /usr/local/lib/ && \
    ldconfig
```

## Enabling RAG in VeriClaw

Add the following to your `~/.vericlaw/config.json` inside the `"tools"` object:

```json
{
  "tools": {
    "rag_enabled": true,
    "rag_embed_base_url": "https://api.openai.com/v1"
  }
}
```

- **`rag_enabled`** – set to `true` to enable the `memory_search` tool and vec extension loading.
- **`rag_embed_base_url`** – base URL for the embeddings endpoint. Defaults to `https://api.openai.com/v1`. For Ollama or a local server use e.g. `http://localhost:11434/v1`.

The API key used for embedding calls is taken from `providers[0].api_key` (your primary provider key).

## Using memory_search

When RAG is enabled, the agent has access to a `memory_search` tool:

```
memory_search(query="what did we discuss about deployment?", k=5)
```

Returns up to `k` (1–10) past memory chunks ranked by vector similarity. The `score` field in results is the raw cosine distance from sqlite-vec (lower = more similar).

## Storing embeddings

Embeddings are **not** stored automatically on every message in the current implementation. You can store a chunk programmatically by calling `Memory.Vector.Store` from Ada code, passing the session ID, content string, and an `Embedding` generated with `Memory.Vector.Embed`.

## Troubleshooting

| Symptom | Fix |
|---|---|
| `memory_search` returns `[]` | sqlite-vec not loaded or tables empty. Check that `rag_enabled: true` and `vec0` is on the library path. |
| HTTP 401 on embed call | API key missing or wrong. Check `providers[0].api_key`. |
| `Embed` returns zeros | HTTP failure calling the embeddings endpoint. Check `rag_embed_base_url` and network access. |
| Extension not found | Set `LD_LIBRARY_PATH` (Linux) or `DYLD_LIBRARY_PATH` (macOS) to the directory containing `vec0.so`/`vec0.dylib`. |
